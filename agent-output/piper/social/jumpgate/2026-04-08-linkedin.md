# Jumpgate LinkedIn — April 8, 2026

## Topic: "Conductor vs. Swarm" frameworks miss the real orchestration problem

---

This week I read three separate articles comparing "conductor" vs. "swarm" architectures for multi-agent AI.

Every one of them focused on coordination patterns: who speaks next, how agents share context, which framework handles routing best.

None of them mentioned the actual hard problem: what happens when Agent A's output changes what Agent B should do next — and neither agent knows it.

We've been running a multi-agent system in production for over a year. Five agents. Shared data layer. Real revenue operations.

Here's what we've learned about orchestration that no framework comparison will tell you:

**1. The coordination problem isn't routing. It's timing.**
Our scout agent finds healthcare leads on Tuesday morning. Our content agent writes a healthcare blog on Wednesday. Our outreach agent sends healthcare-focused emails on Thursday. The orchestration isn't "who talks to whom" — it's "does the content agent know what the scout found before it picks a topic?"

Shared flat files solve this better than any message-passing framework. A markdown file that three agents can read is more reliable than an event bus that two agents might miss.

**2. "Conductor" architectures create a single point of failure.**
We tried centralized orchestration early on. One coordinator agent that told other agents what to do. When it went down for six days (yes, six), nobody noticed — because the other agents had no fallback. They just sat idle waiting for instructions.

Now every agent operates independently and reads shared context. If one agent fails, the others keep working with slightly less information. That's not elegant. It's resilient.

**3. Framework selection is the least important decision you'll make.**
We don't use LangGraph. We don't use CrewAI. We don't use AutoGen. Our agents are cron jobs that read files, make decisions, and write output. The framework is bash + markdown + a scheduler.

The important decisions were: what does each agent own, what data does it share, and what happens when it's wrong? Those decisions survive any framework migration. Your LangGraph routing config won't.

The orchestration discourse is solving the wrong problem. The hard part isn't coordinating agents. It's making sure each agent's work actually improves what the next agent does.

That's a design problem, not a framework problem.
