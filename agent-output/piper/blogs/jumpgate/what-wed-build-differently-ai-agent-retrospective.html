What We Would Build Differently: AI Agent Retrospective

Starting from scratch today with eighteen months of production experience. Specific architectural, process, and team decisions that would change, and a few things we would keep exactly the same.

Jumpgate Industries · February 26, 2026 · 15 min read

Report Highlights

  • • Top AI performers rebuild production agents after an average of 26 months; average performers after 8 months (McKinsey 2025). Starting with the right architecture delays the rebuild cycle significantly.
  • • Three major rebuilds in our first 18 months, each driven by an architectural decision that would have been different with hindsight.
  • • The architectural decision that would change most: starting with modular, separately-deployed agents instead of a monolithic system that grew into modularity through expensive refactoring.
  • • The process decision that would change most: defining handoff protocols for every agent boundary before writing any agent code.
  • • The team decision that would change most: assigning a named owner to each agent before deployment rather than defaulting to collective ownership.
  • • The thing that would stay the same: starting with one agent, one workflow. The temptation to build comprehensively from the start is the single most common cause of first-build failure (LangChain 2026).
  • • Model-agnostic architecture from day one would have reduced year-two maintenance cost by an estimated 70% (Forrester 2025).

The most valuable question in any system retrospective is: knowing what we know now, what would we do differently if we were starting today? At Jumpgate Industries, we have enough production experience to answer this question specifically rather than abstractly. This is the specific list of what would change, why, and what the expected impact would be.

Architectural Changes

Start modular, not monolithic

Our initial system treated the four agents as components of a single, integrated system. Each agent had access to the same codebase, shared configuration, and interoperable state in ways that were convenient during development and created refactoring debt during maintenance. Every change to one agent required understanding its impact on the others. Every rebuild required coordinating changes across the entire system.

Starting today, we would design each agent as an independently deployable unit with a defined public interface: what data it reads, what data it writes, and what events it emits. The agents would coordinate through well-defined contracts rather than shared code. A change to the research agent would require no changes to the outreach agent as long as the contract between them was unchanged. A rebuild of the monitoring agent would not require any downtime for the research or reporting agents.

Design for model changes from day one

When the primary model we built our agents on was updated in month fourteen, significant prompt engineering work was required because the model's behavior had changed in ways that affected agent outputs. Forrester's research found that model-agnostic architecture reduces maintenance costs during model transitions by 70%. Starting today, the model configuration would be a parameter, not a dependency, and every agent's logic would be designed to remain model-neutral in its structure.

Centralized state from day one

We reached centralized state in our second architecture rebuild. We should have started there. The distributed state approach of our first architecture cost more to maintain and debug than the centralized state approach would have cost to build initially. This is not debatable in retrospect. Start with a single source of truth.

Process Changes

Define all handoff protocols before building any agent

LangChain found that teams designing handoff protocols before building any individual agent in a multi-agent system are 2.8 times more likely to reach production. We designed the handoff protocols as we went, updating them as we discovered what the agents actually needed from each other. This produced protocols that were inconsistent, informally documented, and brittle. A complete set of handoff contracts designed before development would have eliminated the second major rebuild entirely.

Month-one monitoring, not month-four monitoring

The monitoring infrastructure was built in month four, after our first major silent failure. It should have been built in month one as a prerequisite for any agent going live. Monitoring is not an operational concern that comes after the agent is working. It is what tells you whether the agent is working. Building it first would not have slowed the development timeline meaningfully. It would have caught the month-four failure in hours instead of days.

Team Changes

Named ownership before deployment

Each agent should have a named owner before it goes live, with that ownership documented alongside the agent's specification. In our initial architecture, ownership was collective, which meant it was effectively no one's. Named ownership from day one would have produced earlier detection of the quality degradation patterns that accumulated in the first six months.

What Would Stay the Same

The single-workflow-first principle. Starting with one agent, one workflow, and making it work before building anything else is the most important decision we made correctly. The temptation to build comprehensively from the start would have produced a system that failed in more places simultaneously and was harder to debug. Narrow scope first was right, and it remains right for every new deployment we do at Jumpgate Industries today.
