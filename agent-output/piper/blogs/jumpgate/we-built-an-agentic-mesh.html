We Built an Agentic Mesh: Architecture Notes from 18 Months of Production

Four coordinated AI agents sharing a central data layer, running 24/7, monitoring 550+ prospects. What the architecture looks like, how it has evolved, and what we learned building it.

An agentic mesh is a multi-agent system where multiple AI agents share a common data layer and operate in coordination, each focused on a specific part of a larger workflow. The term captures the key structural characteristic: the agents are not independent systems running in parallel. They form a connected mesh where the outputs of each agent feed the inputs of others, and all of them draw from and contribute to a shared state that represents the current reality of the pipeline.

Jumpgate Industries has been running an agentic mesh in production for eighteen months. The system covers our full top-of-funnel operation: research, outreach, engagement monitoring, and daily reporting. Four agents, one central data store, running continuously. This post covers the architecture, the evolution, and the specific design decisions that produce stable operation at this level of complexity.

The Architecture

The Central Data Layer

The foundation of the mesh is a single centralized state store. Every agent reads from it. Every agent writes to it. No agent maintains its own copy of shared data. When the research agent deposits a new lead, that lead is immediately available to the outreach agent and the monitoring agent. When the engagement monitoring agent updates a prospect's score, the reporting agent and the outreach agent see the current score on their next cycle. There is no synchronization step and no distributed state divergence.

We learned this lesson the hard way in our first architecture, which had each agent maintaining its own records with a synchronization process running between them. The synchronization process failed in ways that were difficult to detect and difficult to debug. After we consolidated to a single state store, the entire category of synchronization failures disappeared.

The Four Agents

The research agent runs nightly. It scans defined sources for companies matching our qualification criteria, applies the scoring rubric, checks for CRM duplicates, and deposits clean records into the central store. It is the only agent that adds new records to the pipeline.

The outreach agent runs on a regular cycle. It reads qualified leads from the central store, checks whether each one has received initial contact, composes a message for those that have not, and sends it. It updates the contact record in the central store after each action. The outreach agent does not decide which message to send based on a schedule. It reads the engagement signal data from the monitoring agent and adapts timing and content accordingly.

The engagement monitoring agent runs continuously in a lightweight polling loop. It monitors email engagement signals for all active prospects, updates engagement scores in real time, and flags high-engagement prospects for priority follow-up. It does not send messages or deposit leads. It only reads signals and writes scores.

The reporting agent runs every morning before 8am. It reads the current state of the pipeline from the central store and produces a structured brief covering hot prospects, stale threads, pipeline metrics for the past thirty days, and qualified leads waiting for first contact. The brief is delivered to the team's inbox before the workday begins.

Handoff Protocols

Each handoff between agents is governed by an explicit protocol defined before any agent was built. The protocol specifies the fields involved, the timing constraints, the error behavior, and what happens when the expected input is absent. These protocols are the contracts that make the mesh function reliably, and they are the first thing we check when a pipeline anomaly appears.

Mesh Performance Statistics (18 Months)

  • Active prospects monitored: 550+
  • Qualified leads mapped: 460+
  • Open rate: 76% vs. industry 20-25%
  • Reply rate: mid-teens % vs. industry 2-3%
  • Human hours on research and initial outreach: near zero
  • Major architectural rebuilds: 3 (each addressing a different failure class)

What the Three Rebuilds Addressed

Version one used distributed state and broke due to synchronization failures. The second build added centralized state but had implicit data contracts, leading to data format failures when any agent's output structure changed. Version three defined explicit handoff contracts before any agent was built and has been stable since.

The progression confirms the pattern LangChain identified in their 2026 State of Agent Engineering report: 61% of multi-agent failures originate in handoff protocols, not individual agent logic. Our first two rebuilds were both orchestration failures, not agent capability failures. The individual agents were working correctly. The space between them was what broke.
